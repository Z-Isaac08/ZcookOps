## Contexte & Architecture

| Paramètre            | Valeur               |
| -------------------- | -------------------- |
| **Serveur**          | `dc1.lab.local`      |
| **OS**               | Debian 12 Bookworm   |
| **Adresse IP**       | `192.168.56.10 / 24` |
| **Domaine LDAP**     | `dc=lab,dc=local`    |
| **DNS**              | BIND9                |
| **DHCP**             | isc-dhcp-server      |
| **Annuaire**         | OpenLDAP (slapd)     |
| **Authentification** | SSSD + NSS + PAM     |
| **Accès distant**    | OpenSSH (port 2222)  |
| **Hyperviseur**      | VMware Workstation   |

**Objectif :** Déployer une infrastructure réseau complète basée sur un contrôleur de domaine Linux, permettant la gestion centralisée des identités (utilisateurs, groupes, machines) et des services réseau (DNS, DHCP, SSH) en utilisant exclusivement des technologies open source.

**Ordre de déploiement obligatoire :**

```
Réseau → DNS (BIND9) → OpenLDAP → DHCP → SSH + SSSD → Sauvegarde
```

> [!NOTE]
> BIND9 doit être configuré **avant** OpenLDAP. Le daemon `slapd` utilise le FQDN `dc1.lab.local` lors de son initialisation — ce nom doit être résolvable localement au moment de l'installation.

## 1. Glossaire des Services

Avant de commencer, voici le rôle de chaque composant et la raison de son choix.

### LDAP & OpenLDAP

**LDAP** (Lightweight Directory Access Protocol, RFC 4511) est un protocole TCP/IP permettant d'accéder à un annuaire d'informations hiérarchique. Les objets (utilisateurs, groupes, machines) sont organisés en arbre appelé **DIT** (Directory Information Tree). Chaque entrée est identifiée par un **DN** (Distinguished Name) unique.

| Caractéristique   | Valeur                              |
| ----------------- | ----------------------------------- |
| Port standard     | 389 (LDAP) / 636 (LDAPS)            |
| Implémentation    | OpenLDAP (slapd)                    |
| Format de données | LDIF (LDAP Data Interchange Format) |

**OpenLDAP** est l'implémentation open source de référence. Contrairement à FreeIPA qui impose une structure prédéterminée, OpenLDAP offre un contrôle total sur les ACL, les schémas d'objets et la structure du DIT.

### DNS & BIND9

Le DNS traduit les noms lisibles (`dc1.lab.local`) en adresses IP (`192.168.56.10`) et inversement. Deux types de zones sont utilisées :

- **Zone directe** : nom → IP (enregistrement `A`)
- **Zone inverse** : IP → nom (enregistrement `PTR`)
- **Forwarders** : si le nom est externe (`google.com`), délégation vers `8.8.8.8`

### DHCP

Le DHCP (RFC 2131) distribue automatiquement les adresses IP aux clients à leur connexion. Le flux est `DISCOVER → OFFER → REQUEST → ACK`. Il transmet également l'adresse DNS (`192.168.56.10`) aux clients, garantissant la résolution automatique du domaine `lab.local`.

### SSSD, PAM & NSS

Ces trois composants forment la couche d'authentification Linux-LDAP :

| Composant            | Rôle                                                                                                                     |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **SSSD**             | Middleware entre le système Linux et LDAP. Cache les credentials en local pour survivre à une panne réseau.              |
| **PAM**              | Couche basse de l'authentification Linux. SSH, login et sudo délèguent la vérification des identités à PAM.              |
| **NSS**              | Définit l'ordre de résolution des identités. En ajoutant `sss` dans `nsswitch.conf`, le système interroge LDAP via SSSD. |
| **libpam-sss**       | Module PAM vers SSSD                                                                                                     |
| **libnss-sss**       | Module NSS vers SSSD                                                                                                     |
| **oddjob-mkhomedir** | Crée `/home/utilisateur` automatiquement à la première connexion                                                         |

### SSH

SSH (RFC 4251) établit des sessions distantes chiffrées. Le port par défaut (22) est remplacé par **2222** pour réduire l'exposition aux scans automatisés et aux attaques brute-force. Grâce à SSSD et PAM, les utilisateurs LDAP se connectent en SSH sans compte local sur la machine.

## 2. Mise en place de l'Infrastructure

### 2.1 Configuration VMware Workstation

| Ressource      | Valeur recommandée                  |
| -------------- | ----------------------------------- |
| RAM            | 2 GB                                |
| vCPU           | 2                                   |
| Disque OS      | 20 GB                               |
| Carte réseau 1 | NAT (VMnet8) — accès internet       |
| Carte réseau 2 | Host-Only (VMnet1) — réseau interne |

> [!CAUTION]
> Désactiver le DHCP intégré de VMware sur le réseau Host-Only (VMnet1). C'est le serveur `dc1` qui assure ce rôle.

### 2.2 Installation de Debian 12

Installation en mode **headless** (sans environnement graphique). Paramètres de l'installateur :

| Étape installateur  | Valeur                                     |
| ------------------- | ------------------------------------------ |
| Hostname            | `dc1`                                      |
| Domain name         | `lab.local`                                |
| Paquets (tasksel)   | SSH server + Standard utilities uniquement |
| Desktop environment | **Ne pas installer**                       |

**Partitionnement :**

| Partition | Taille | Point de montage | Type                       |
| --------- | ------ | ---------------- | -------------------------- |
| sda1      | 9,3 GB | `/`              | ext4                       |
| sda2      | 4,7 GB | `/var`           | ext4 (logs + données LDAP) |
| sda3      | 1,9 GB | [SWAP]           | swap                       |
| sda4      | 4,2 GB | `/backup`        | ext4 (sauvegardes)         |

### 2.3 Configuration de l'Adresse IP Statique

Identifier les interfaces disponibles :

```bash
ip a
```

Éditer le fichier de configuration réseau :

```bash
nano /etc/network/interfaces
```

```bash
# Loopback
auto lo
iface lo inet loopback

# NAT - accès internet (DHCP automatique)
auto ens33
iface ens33 inet dhcp

# Host-Only - réseau interne (IP fixe)
auto ens34
iface ens34 inet static
    address 192.168.56.10
    netmask 255.255.255.0
```

> [!CAUTION]
> Ne pas définir de passerelle sur `ens34`. La passerelle par défaut reste sur `ens33` (NAT). Les deux interfaces sont indépendantes.

Appliquer la configuration :

```bash
systemctl restart networking
```

### 2.4 Configuration du Hostname et de /etc/hosts

Vérifier que le FQDN est correctement configuré :

```bash
hostname -f
# Résultat attendu : dc1.lab.local
```

Si ce n'est pas le cas, corriger manuellement :

```bash
hostnamectl set-hostname dc1
nano /etc/hosts
```

Contenu de `/etc/hosts` :

```
127.0.0.1   localhost
192.168.56.10   dc1.lab.local dc1
```

> [!WARNING]
> Supprimer toute ligne associant `dc1` à `127.0.1.1`. OpenLDAP et BIND9 utilisent ce FQDN dans leur configuration — il doit pointer vers l'IP réelle du serveur, pas le loopback.

apt update && apt upgrade -y

apt install -y \

| Paquet           | Rôle                                            |
| ---------------- | ----------------------------------------------- |
| `net-tools`      | `ifconfig`, `netstat` pour le diagnostic réseau |
| `dnsutils`       | `dig`, `nslookup` pour tester le DNS            |

## 3. DNS avec BIND9

### 3.1 Installation

```bash
apt install -y bind9 bind9utils bind9-doc
systemctl status bind9
```

### 3.2 Configuration Principale — named.conf.options

Ce fichier définit les paramètres globaux : interfaces d'écoute, forwarders et contrôle d'accès.

```bash
nano /etc/bind/named.conf.options
```

```bind
options {
    directory "/var/cache/bind";

    # Si BIND9 ne connaît pas le nom, il délègue à Google DNS
    forwarders {
        8.8.8.8;
        8.8.4.4;
    };

    dnssec-validation auto;

    # Écouter uniquement sur les interfaces utiles
    listen-on { 127.0.0.1; 192.168.56.10; };

    # Autoriser uniquement le réseau interne
    allow-query { localhost; 192.168.56.0/24; };

    # Activer la résolution récursive pour les clients
    recursion yes;
};
```

### 3.3 Déclaration des Zones — named.conf.local

```bash
nano /etc/bind/named.conf.local
```

```bind
# Zone directe : lab.local → IP
zone "lab.local" {
    type master;
    file "/etc/bind/zones/db.lab.local";
};

# Zone inverse : IP → lab.local
zone "56.168.192.in-addr.arpa" {
    type master;
    file "/etc/bind/zones/db.192.168.56";
};
```

```bash
mkdir /etc/bind/zones
```

### 3.4 Zone Directe

```bash
nano /etc/bind/zones/db.lab.local
```

```bind
$TTL 604800
@ IN SOA dc1.lab.local. admin.lab.local. (
        2024010101 ; Serial
        604800     ; Refresh
        86400      ; Retry
        2419200    ; Expire
        604800 )   ; Negative Cache TTL

@ IN NS  dc1.lab.local.
dc1 IN A 192.168.56.10
```

### 3.5 Zone Inverse

```bash
nano /etc/bind/zones/db.192.168.56
```

```bind
$TTL 604800
@ IN SOA dc1.lab.local. admin.lab.local. (
        2024010101 ; Serial
        604800     ; Refresh
        86400      ; Retry
        2419200    ; Expire
        604800 )   ; Negative Cache TTL

@ IN NS  dc1.lab.local.
10 IN PTR dc1.lab.local.
```

### 3.6 Validation et Démarrage

Vérifier la syntaxe avant tout redémarrage :

```bash
named-checkconf
named-checkzone lab.local /etc/bind/zones/db.lab.local
named-checkzone 56.168.192.in-addr.arpa /etc/bind/zones/db.192.168.56
```

```bash
systemctl unmask bind9    # Nécessaire sur Debian 12 (lien symbolique)
systemctl enable --now bind9
```

Pointer le serveur vers son propre DNS :

```bash
nano /etc/resolv.conf
```

```
domain lab.local
search lab.local
nameserver 192.168.56.10
```

Tests de validation :

```bash
dig dc1.lab.local
dig -x 192.168.56.10
nslookup dc1.lab.local 192.168.56.10
```

## 4. DHCP avec isc-dhcp-server

### 4.1 Installation

```bash
apt install -y isc-dhcp-server
```

Définir l'interface d'écoute (réseau interne uniquement) :

```bash
nano /etc/default/isc-dhcp-server
```

```bash
INTERFACESv4="ens34"
```

> [!CAUTION]
> Ne jamais mettre `ens33` (NAT). Le serveur DHCP ne doit distribuer des adresses que sur le réseau interne Host-Only.

### 4.2 Configuration dhcpd.conf

```bash
nano /etc/dhcp/dhcpd.conf
```

```bash
default-lease-time 600;
max-lease-time 7200;
authoritative;

# Domaine et DNS transmis aux clients
option domain-name "lab.local";
option domain-name-servers 192.168.56.10;

subnet 192.168.56.0 netmask 255.255.255.0 {
    range 192.168.56.100 192.168.56.200;
    option routers 192.168.56.10;
    option broadcast-address 192.168.56.255;
}
```

| Directive             | Rôle                                           |
| --------------------- | ---------------------------------------------- |
| `authoritative`       | Ce serveur fait autorité sur ce réseau         |
| `domain-name-servers` | DNS communiqué aux clients (dc1 lui-même)      |
| `range`               | Plage d'adresses distribuées : `.100` à `.200` |
| `option routers`      | Passerelle par défaut transmise aux clients    |
| `default-lease-time`  | Durée du bail en secondes (600 = 10 minutes)   |

### 4.3 Démarrage et Tests

```bash
systemctl restart isc-dhcp-server
systemctl enable isc-dhcp-server
systemctl status isc-dhcp-server
```

## 5. Annuaire OpenLDAP

OpenLDAP est le cœur de l'infrastructure. Il stocke l'ensemble des identités du domaine `lab.local` et répond aux requêtes d'authentification de toutes les machines configurées avec SSSD.

### 5.1 Installation

```bash
apt install -y slapd ldap-utils
```

Reconfigurer le domaine LDAP après installation :

```bash
dpkg-reconfigure slapd
```

| Question de l'installateur          | Réponse                          |
| ----------------------------------- | -------------------------------- |
| Omit OpenLDAP server configuration? | Non                              |
| DNS domain name?                    | `lab.local`                      |
| Organization name?                  | `lab`                            |
| Administrator password?             | (mot de passe fort, à conserver) |
| Remove the database when purged?    | Non                              |
| Move old database?                  | Oui                              |

Vérifier l'installation :

```bash
ldapsearch -x -b "dc=lab,dc=local" -H ldap://localhost
# Résultat attendu : result: 0 Success avec numEntries: 1
```

### 5.2 Structure du DIT (Directory Information Tree)

L'annuaire est organisé en arbre hiérarchique :

```
dc=lab,dc=local                    (racine du domaine)
 ├── ou=Users                      (utilisateurs)
 │    ├── uid=alice
 │    └── uid=bob
 ├── ou=Groups                     (groupes)
 │    ├── cn=admins
 │    └── cn=users
 └── ou=Computers                  (machines)
      ├── cn=dc1
      └── cn=client1
```

### 5.3 Création des Unités Organisationnelles

```bash
nano /etc/ldap/structure.ldif
```

```ldif
dn: ou=Users,dc=lab,dc=local
objectClass: organizationalUnit
ou: Users

dn: ou=Groups,dc=lab,dc=local
objectClass: organizationalUnit
ou: Groups

dn: ou=Computers,dc=lab,dc=local
objectClass: organizationalUnit
ou: Computers
```

```bash
ldapadd -x -D "cn=admin,dc=lab,dc=local" -W -f /etc/ldap/structure.ldif
```

| Option `ldapadd` | Signification                                  |
| ---------------- | ---------------------------------------------- |
| `-x`             | Authentification simple (sans Kerberos)        |
| `-D`             | DN de l'administrateur LDAP                    |
| `-W`             | Demande le mot de passe de manière interactive |
| `-f`             | Fichier LDIF à injecter                        |

### 5.4 Création des Utilisateurs

Générer les mots de passe hashés avant de créer les utilisateurs :

```bash
slappasswd
```

> [!NOTE]
> `slappasswd` génère un hash SSHA (Salted SHA). Le sel aléatoire garantit que deux mots de passe identiques produisent des hashs différents. **Ne jamais stocker de mots de passe en clair dans LDAP.**

```bash
nano /etc/ldap/users.ldif
```

```ldif
dn: uid=alice,ou=Users,dc=lab,dc=local
objectClass: inetOrgPerson
objectClass: posixAccount
objectClass: shadowAccount
uid: alice
cn: Alice Dupont
sn: Dupont
givenName: Alice
loginShell: /bin/bash
homeDirectory: /home/alice
uidNumber: 1001
gidNumber: 1001
userPassword: {SSHA}HashGenereParSlappasswd
mail: alice@lab.local

dn: uid=bob,ou=Users,dc=lab,dc=local
objectClass: inetOrgPerson
objectClass: posixAccount
objectClass: shadowAccount
uid: bob
cn: Bob Martin
sn: Martin
givenName: Bob
loginShell: /bin/bash
homeDirectory: /home/bob
uidNumber: 1002
gidNumber: 1002
userPassword: {SSHA}HashGenereParSlappasswd
mail: bob@lab.local
```

| Attribut LDIF                | Rôle                                                 |
| ---------------------------- | ---------------------------------------------------- |
| `objectClass: posixAccount`  | Permet la connexion Linux (UID, GID, shell, home)    |
| `objectClass: shadowAccount` | Gestion des politiques de mots de passe (expiration) |
| `uidNumber / gidNumber`      | Identifiants Unix uniques, obligatoirement > 1000    |
| `loginShell`                 | Shell attribué à la connexion                        |
| `userPassword`               | Mot de passe hashé en SSHA via `slappasswd`          |

```bash
ldapadd -x -D "cn=admin,dc=lab,dc=local" -W -f /etc/ldap/users.ldif
```

### 5.5 Création des Groupes

```bash
nano /etc/ldap/groups.ldif
```

```ldif
dn: cn=admins,ou=Groups,dc=lab,dc=local
objectClass: posixGroup
cn: admins
gidNumber: 1001
memberUid: alice

dn: cn=users,ou=Groups,dc=lab,dc=local
objectClass: posixGroup
cn: users
gidNumber: 1002
memberUid: alice
memberUid: bob
```

```bash
ldapadd -x -D "cn=admin,dc=lab,dc=local" -W -f /etc/ldap/groups.ldif
```

### 5.6 Enregistrement des Machines

```bash
nano /etc/ldap/computers.ldif
```

```ldif
dn: cn=dc1,ou=Computers,dc=lab,dc=local
objectClass: device
objectClass: top
cn: dc1
description: Controleur de domaine

dn: cn=client1,ou=Computers,dc=lab,dc=local
objectClass: device
objectClass: top
cn: client1
description: Poste client Linux
```

```bash
ldapadd -x -D "cn=admin,dc=lab,dc=local" -W -f /etc/ldap/computers.ldif
```

### 5.7 Configuration des ACL

Les ACL protègent les attributs sensibles, notamment les mots de passe.

```bash
nano /etc/ldap/acl.ldif
```

```ldif
dn: olcDatabase={1}mdb,cn=config
changetype: modify
replace: olcAccess
olcAccess: {0}to attrs=userPassword
    by self write
    by anonymous auth
    by dn="cn=admin,dc=lab,dc=local" write
    by * none
olcAccess: {1}to attrs=shadowLastChange
    by self write
    by * read
olcAccess: {2}to *
    by dn="cn=admin,dc=lab,dc=local" write
    by users read
    by * none
```

```bash
ldapmodify -Y EXTERNAL -H ldapi:/// -f /etc/ldap/acl.ldif
```

| Règle ACL              | Signification                                                   |
| ---------------------- | --------------------------------------------------------------- |
| `{0} userPassword`     | Mot de passe : visible uniquement par l'utilisateur et l'admin  |
| `{1} shadowLastChange` | Date de changement de MDP : lisible par tous                    |
| `{2} *`                | Tous les autres attributs : lisibles par les users authentifiés |

### 5.8 Vérification

```bash
# Vérifier toute la structure
ldapsearch -x -b "dc=lab,dc=local" -H ldap://localhost

# Tester l'authentification d'un utilisateur
ldapwhoami -x -D "uid=alice,ou=Users,dc=lab,dc=local" -W
```

Résultat attendu pour `ldapsearch` : `numEntries: 8` ou plus (racine + 3 OU + utilisateurs + groupes + machines).

## 6. Sécurisation SSH

### 6.1 Durcissement de sshd_config

```bash
nano /etc/ssh/sshd_config
```

```bash
# Changer le port pour réduire les attaques automatisées
Port 2222

# Interdire la connexion root directe
PermitRootLogin no

# Activer l'authentification par mot de passe
PasswordAuthentication yes

# Activer l'authentification par clé publique
PubkeyAuthentication yes

# Limiter les tentatives d'authentification
MaxAuthTries 3

# Déconnecter les sessions inactives après 5 minutes
ClientAliveInterval 300
ClientAliveCountMax 2

# Restreindre l'accès aux groupes autorisés
AllowGroups sudo users admins
```

| Directive SSH             | Impact sécurité                                                |
| ------------------------- | -------------------------------------------------------------- |
| `Port 2222`               | Réduit les scans automatiques sur le port standard 22          |
| `PermitRootLogin no`      | Empêche les attaques brute-force directes sur root             |
| `MaxAuthTries 3`          | Limite les tentatives avant déconnexion forcée                 |
| `ClientAliveInterval 300` | Ferme les sessions SSH abandonnées après 5 minutes             |
| `AllowGroups`             | Seuls les membres des groupes LDAP listés peuvent se connecter |

```bash
systemctl restart ssh
```

> [!WARNING]
> Ne pas fermer la session SSH active avant d'avoir validé la connexion sur le nouveau port depuis un second terminal.

### 6.2 Test de Connexion

```bash
ssh -p 2222 alice@192.168.56.10
```

## 7. Intégration LDAP avec Linux (SSSD)

SSSD fait le pont entre le système Linux et l'annuaire OpenLDAP. Il permet aux utilisateurs LDAP de se connecter comme s'ils étaient des utilisateurs locaux.

**Flux d'authentification :**

```
alice tente de se connecter (SSH ou console)
        │
        ▼
PAM intercepte la demande
        │
        ▼
SSSD interroge OpenLDAP sur dc1
        │
        ▼
OpenLDAP valide les credentials d'alice
        │
        ▼
SSSD retourne OK à PAM
        │
        ▼
Connexion autorisée + /home/alice créé si absent
```

### 7.1 Installation

```bash
apt install -y sssd sssd-ldap libpam-sss libnss-sss oddjob-mkhomedir
```

### 7.2 Configuration de sssd.conf

```bash
nano /etc/sssd/sssd.conf
```

```ini
[sssd]
services = nss, pam
domains = lab.local

[domain/lab.local]
id_provider = ldap
auth_provider = ldap
ldap_uri = ldap://192.168.56.10
ldap_search_base = dc=lab,dc=local
ldap_user_search_base = ou=Users,dc=lab,dc=local
ldap_group_search_base = ou=Groups,dc=lab,dc=local
ldap_default_bind_dn = cn=admin,dc=lab,dc=local
ldap_default_authtok = MotDePasseAdmin
ldap_tls_reqcert = never
ldap_id_use_start_tls = false
cache_credentials = true
enumerate = true
```

Sécuriser le fichier (SSSD refuse de démarrer si les permissions sont trop ouvertes) :

```bash
chmod 600 /etc/sssd/sssd.conf
```

### 7.3 Modification de nsswitch.conf

```bash
nano /etc/nsswitch.conf
```

Ajouter `sss` comme source d'identités après les fichiers locaux :

```
passwd: files sss
group:  files sss
shadow: files sss
```

### 7.4 Configuration PAM

Activer la création automatique du répertoire home à la première connexion d'un utilisateur LDAP :

```bash
pam-auth-update --enable mkhomedir
```

### 7.5 Démarrage et Tests

```bash
# Vider le cache avant le premier démarrage
systemctl stop sssd
rm -rf /var/lib/sss/db/*
systemctl start sssd
systemctl enable sssd
```

> [!NOTE]
> Vider le cache SSSD avant le premier démarrage évite les problèmes de credentials obsolètes issus d'une configuration précédente.

Vérifications :

```bash
# Vérifier qu'alice est visible depuis le système
id alice

# Lister tous les utilisateurs du domaine
getent passwd

# Tester la connexion SSH avec un utilisateur LDAP
ssh -p 2222 alice@192.168.56.10
```

Résultat attendu pour `id alice` :

```
uid=1001(alice) gid=1001(admins) groupes=1001(admins),1002(users)
```

## 8. Maintenance et Sauvegarde

Les sauvegardes sont stockées sur la partition `/backup` (sda4) dédiée à cet usage.

### 8.1 Architecture de Sauvegarde

```
/backup  (partition sda4 - 4 GB)
 ├── ldap/      ← exports LDIF de l'annuaire OpenLDAP
 ├── configs/   ← archives des fichiers de configuration
 └── logs/      ← journaux d'exécution des sauvegardes
```

```bash
mkdir -p /backup/{ldap,configs,logs}
```

### 8.2 Export LDAP avec slapcat

`slapcat` exporte l'intégralité de la base OpenLDAP au format LDIF. Contrairement à `ldapsearch`, il lit directement les fichiers de base de données et ne nécessite pas que `slapd` soit en cours d'exécution.

```bash
slapcat -v -l /backup/ldap/ldap_manuel_$(date +%Y%m%d).ldif
```

| Option `slapcat` | Rôle                                         |
| ---------------- | -------------------------------------------- |
| `-v`             | Mode verbeux : affiche les entrées exportées |
| `-l`             | Fichier de sortie au format LDIF             |

### 8.3 Script de Sauvegarde Automatisé

```bash
nano /usr/local/bin/backup.sh
```

```bash
#!/bin/bash
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/backup"
LOG="$BACKUP_DIR/logs/backup_$DATE.log"

echo "=== Sauvegarde du $DATE ===" | tee $LOG

# 1. Sauvegarde LDAP
echo "[1/5] Sauvegarde LDAP..." | tee -a $LOG
slapcat -v -l $BACKUP_DIR/ldap/ldap_$DATE.ldif 2>> $LOG
echo "OK LDAP sauvegardé" | tee -a $LOG

# 2. Sauvegarde DNS
echo "[2/5] Sauvegarde DNS..." | tee -a $LOG
tar -czf $BACKUP_DIR/configs/bind_$DATE.tar.gz /etc/bind/ 2>> $LOG
echo "OK DNS sauvegardé" | tee -a $LOG

# 3. Sauvegarde DHCP
echo "[3/5] Sauvegarde DHCP..." | tee -a $LOG
tar -czf $BACKUP_DIR/configs/dhcp_$DATE.tar.gz /etc/dhcp/ 2>> $LOG
echo "OK DHCP sauvegardé" | tee -a $LOG

# 4. Sauvegarde SSH
echo "[4/5] Sauvegarde SSH..." | tee -a $LOG
tar -czf $BACKUP_DIR/configs/ssh_$DATE.tar.gz /etc/ssh/ 2>> $LOG
echo "OK SSH sauvegardé" | tee -a $LOG

# 5. Sauvegarde SSSD
echo "[5/5] Sauvegarde SSSD..." | tee -a $LOG
tar -czf $BACKUP_DIR/configs/sssd_$DATE.tar.gz /etc/sssd/ 2>> $LOG
echo "OK SSSD sauvegardé" | tee -a $LOG

# Rotation : conserver uniquement les 7 derniers jours
find $BACKUP_DIR/ldap/    -name "*.ldif"   -mtime +7 -delete
find $BACKUP_DIR/configs/ -name "*.tar.gz" -mtime +7 -delete
find $BACKUP_DIR/logs/    -name "*.log"    -mtime +7 -delete

echo "=== Sauvegarde terminée ===" | tee -a $LOG
```

```bash

```

### 9.2 Tableau des Erreurs Courantes

| Erreur                                    | Cause probable               | Solution                                |
| ----------------------------------------- | ---------------------------- | --------------------------------------- |
| `result: 32 No such object`               | OU inexistante dans LDAP     | Créer l'OU manquante avec `ldapadd`     |
| `Invalid credentials (49)`                | Mauvais mot de passe LDAP    | Réinitialiser avec `ldappasswd`         |
| `Backend is offline (SSSD)`               | TLS non supporté             | Ajouter `ldap_id_use_start_tls = false` |
| `Refusing to operate on linked unit file` | Lien symbolique systemd      | Exécuter `systemctl unmask bind9`       |
| `Permission denied SSH (code 6)`          | User absent de `AllowGroups` | Ajouter le groupe dans `sshd_config`    |
| `Connection timed out SSH`                | Mauvais VMnet ou firewall    | Vérifier VMnet, `ping`, `ufw status`    |
| `SSSD ne trouve pas les users`            | Cache corrompu               | `rm -rf /var/lib/sss/db/*` puis restart |

### 9.3 Déverrouillage d'un Compte LDAP

Un compte peut être verrouillé après plusieurs échecs de connexion. L'attribut `pwdAccountLockedTime` est alors positionné.

```bash

```

```ldif
dn: uid=alice,ou=Users,dc=lab,dc=local
changetype: modify
delete: pwdAccountLockedTime
```

```bash
ldapmodify -x -D "cn=admin,dc=lab,dc=local" -W -f /tmp/unlock.ldif
```

Réinitialiser le mot de passe d'un utilisateur :

```bash

    "uid=alice,ou=Users,dc=lab,dc=local"
```

## Conclusion

Ce projet démontre qu'il est possible de construire une infrastructure de type **Contrôleur de Domaine Linux** complète et sécurisée en utilisant exclusivement des solutions open source.

**Chaîne de déploiement :**

```
1. VMware       → Configuration des cartes réseau (NAT + Host-Only)
2. Debian 12    → Installation headless + partitionnement dédié
3. Réseau       → IP statique, hostname FQDN, /etc/hosts
4. BIND9        → DNS interne lab.local (zones directe + inverse)
5. OpenLDAP     → DIT : Users, Groups, Computers + ACL
6. DHCP         → Distribution IP + DNS aux clients
7. SSH          → Durcissement port 2222 + AllowGroups LDAP
8. SSSD+PAM+NSS → Pont LDAP ↔ système Linux
9. Cron/slapcat → Sauvegarde quotidienne automatisée
```

**Services déployés :**

| Service    | Paquet Debian                      | Rôle                                |
| ---------- | ---------------------------------- | ----------------------------------- |
| DNS        | `bind9`                            | Résolution de noms pour `lab.local` |
| DHCP       | `isc-dhcp-server`                  | Distribution d'adresses IP          |
| Annuaire   | `slapd`, `ldap-utils`              | Base d'identités centralisée        |
| Auth Linux | `sssd`, `libpam-sss`, `libnss-sss` | Pont LDAP ↔ système Linux           |
| SSH        | `openssh-server`                   | Accès distant sécurisé (port 2222)  |

Cette architecture offre une alternative solide et flexible aux solutions propriétaires (Active Directory) pour la gestion centralisée d'un parc informatique Linux.
