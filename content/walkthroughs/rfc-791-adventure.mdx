## L'Aventure RFC 791

Lire une RFC, c'est comme remonter aux sources d'une civilisation. La **RFC 791**, publiée en 1981, définit le protocole IP (Internet Protocol). C'est ici que tout commence.

### 1. Pourquoi lire la RFC 791 aujourd'hui ?
Même si nous utilisons massivement l'IPv6, comprendre l'IPv4 via sa spécification d'origine permet de saisir les contraintes et les génies d'ingénierie de l'époque.

### 2. L'En-tête IP (Header)
C'est la partie la plus dense. Chaque bit compte.

| Field | Size | Description |
| :--- | :--- | :--- |
| **Version** | 4 bits | Toujours 4 pour l'IPv4. |
| **IHL** | 4 bits | Internet Header Length. |
| **Type of Service** | 8 bits | Priorité du paquet. |
| **Total Length** | 16 bits | Taille totale du paquet (header + data). |

> [!TIP]
> L'IHL est crucial car il indique où commencent les données. Sans lui, impossible de parser le paquet si des options sont présentes.

### 3. La Fragmentation
Un aspect souvent mal compris. La RFC explique que si un paquet est trop grand pour un réseau physique (MTU), il doit être fragmenté.
- **Identification** : Pour regrouper les fragments.
- **Flags** : `DF` (Don't Fragment) et `MF` (More Fragments).
- **Fragment Offset** : Position du fragment dans le datagramme original.

### 4. Mes Réflexions
Ce qui me frappe, c'est la simplicité apparente qui cache une robustesse incroyable. Le protocole est conçu pour être "best-effort", sans garantie de livraison, laissant cette responsabilité aux couches supérieures (TCP).
